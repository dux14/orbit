Fecha: 2026-06-06 · Proyecto Supabase: `vmcjkleuetcogqhdnlfx`.

## 1. Seeding de usuario/sesión para E2E de sync

Los specs `e2e/multi-device.spec.ts` y `e2e/sync.spec.ts` NO usan un usuario
estático ni un `TEST_SUPABASE_SESSION` exportado (desviación deliberada del plan,
establecida en S8): cada test **crea un usuario throwaway por run** vía la REST
API de Supabase y le inyecta la sesión como cookie de `@supabase/ssr`.

- Signup: `POST {SUPABASE_URL}/auth/v1/signup` con la anon key →
  email único `e2e-<proyecto>-<timestamp>@orbit-e2e.example.com`. Devuelve la
  sesión completa (requiere `mailer_autoconfirm=true`, ver §1.1).
- Inyección: cookie `sb-<ref>-auth-token` = `base64-` + base64url(JSON de la
  sesión), chunks de 3180 chars (`.0`, `.1`, …) — espejo exacto del storage de
  `createBrowserClient`. Helpers en `e2e/helpers/multi-device.ts`.
- Aislamiento de dispositivo: un `BrowserContext` = un dispositivo (IndexedDB,
  localStorage y cookies aislados), misma cuenta en todos.
- Teardown: cada test borra su fila de `vaults` (RLS permite borrar la propia).
  Los usuarios `e2e-%@orbit-e2e.example.com` se limpian periódicamente por SQL
  (Management API):
  ```sql
  delete from auth.users where email like 'e2e-%@orbit-e2e.example.com';
  ```
- Ejecución: `NEXT_PUBLIC_SYNC_ENABLED=true pnpm test:e2e` (la flag se inlinea
  en la build del webServer; si hay un server previo en :3000 sin la flag,
  matarlo primero — `reuseExistingServer` lo reutilizaría sin sync).

Ventaja sobre el `TEST_SUPABASE_SESSION` estático del plan: cero credenciales
persistentes (ni en `.env.local` ni en CI), sin estado compartido entre runs,
y sin riesgo de expiración del token sembrado.

### 1.1 Dependencia: `mailer_autoconfirm=true`

El signup por API devuelve sesión solo porque el proyecto tiene
`mailer_autoconfirm=true`. Riesgo documentado en S11 (`s11-rls.md` §2.2):
signups por API sin verificación de email. **Decisión pendiente pre-producción**
(desactivar / captcha / deshabilitar provider email — rompería este harness;
la alternativa sería acuñar sesiones con la service key en CI).
